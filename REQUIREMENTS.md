# PayCore — Requirements Specification

> Этап 1 жизненного цикла ПО: анализ требований.
> Учебный production-grade проект: платформа цифрового банка на микросервисах (Go).
> Этот документ — источник истины для домена и нефункциональных требований. Каждое архитектурное решение в проекте должно быть прослеживаемо до требования отсюда.

---

## 1. Видение и цель

PayCore — мини-необанк: пользователи держат мультивалютные кошельки, пополняют их, переводят деньги друг другу (в том числе между разными валютами) и выводят средства. Каждое движение денег — неизменяемая двойная запись в леджере; межсервисные операции выполняются как saga с компенсациями.

Цель проекта — не «работающий перевод», а отработка production-практик распределённых систем на Go: консистентность под параллелью, надёжная доставка событий, отказоустойчивость, наблюдаемость и доказуемая корректность.

---

## 2. Объём (scope)

### В объёме
- Регистрация/аутентификация, JWT (access + refresh).
- Профиль клиента, статус KYC, лимиты.
- Мультивалютные кошельки, двойная запись, балансы.
- Пополнение и вывод (симулированные — без реальных банковских рельсов).
- Перевод между кошельками: одновалютный и кросс-валютный, через saga.
- Синхронная проверка риска (fraud/velocity) в критическом пути.
- Конвертация валют по курсам (FX-провайдер мокается).
- История операций как отдельная read-модель (CQRS).
- Уведомления и неизменяемый аудит как асинхронные консьюмеры.
- Reconciliation: периодическая сверка балансов и поиск зависших saga.

### Вне объёма
- Реальная интеграция с банками/платёжными системами.
- Реальный KYC-провайдер (статус выставляется вручную/стабом).
- Реальный FX-провайдер (курсы из стаба/сидов).
- Мобильный/веб-клиент (только API).
- Мультирегиональность, гео-репликация.

## 3. Единый язык домена (ubiquitous language)

| Термин | Значение |
|---|---|
| Customer | Пользователь платформы; владелец кошельков. |
| Account | Учётная запись клиента: профиль, KYC-статус, лимиты. Не путать с бухгалтерским счётом. |
| Wallet | Кошелёк клиента в конкретной валюте. У клиента может быть несколько. |
| Ledger | Бухгалтерская книга: неизменяемый журнал проводок. |
| Ledger entry (posting) | Одна сторона проводки (дебет или кредит) по конкретному кошельку. |
| Transaction | Логическая операция, состоящая из сбалансированного набора проводок (Σ debit = Σ credit). |
| Balance | Производная величина: результат суммирования проводок кошелька. |
| Transfer | Перемещение средств между двумя кошельками; реализуется как saga. |
| Saga | Распределённая транзакция как последовательность локальных шагов с компенсациями. |
| Idempotency-Key | Клиентский ключ, гарантирующий единичный эффект при повторе запроса. |
| Outbox | Таблица событий, пишется в одной транзакции с бизнес-данными; основа надёжной публикации. |
| Inbox | Таблица обработанных событий на стороне консьюмера; основа дедупликации. |
| Risk decision | Решение Fraud-сервиса: allow / block + причина. |
| Reconciliation | Сверка: пересчёт балансов из проводок и поиск расхождений/зависших saga. |

## 4. Границы контекстов → сервисы

| Bounded context | Сервис | Ответственность |
|---|---|---|
| Identity & Access | Auth | Регистрация, логин, выпуск/ротация/отзыв токенов. |
| Customer | Account | Профиль, KYC-статус, лимиты. |
| Ledger | Wallet/Ledger | Кошельки, проводки, балансы, идемпотентные постинги. |
| Payments | Payment Orchestrator | Saga перевода, состояние, идемпотентность, компенсации. |
| Risk | Fraud/Risk | Синхронная оценка риска, velocity-лимиты. |
| Pricing | FX | Курсы, конвертация, кэш. |
| Reporting | History (CQRS) | Read-модель истории операций из событий. |
| Messaging | Notification | Доставка уведомлений (стаб). |
| Compliance | Audit | Неизменяемый журнал всех доменных событий. |

Каждый контекст владеет своей БД. Общение — только через API (gRPC) и события (Kafka). Общего стораджа нет.

---

## 5. Акторы

- **Customer** — конечный пользователь (через API).
- **Compliance officer / Admin** — читает аудит, управляет KYC и лимитами (упрощённо).
- **System / Scheduler** — фоновые процессы: outbox relay, reconciliation, ротация токенов.
- **External FX provider** — внешний источник курсов (мокается).

---

## 6. Функциональные требования

### Identity & Access (Auth)
- **FR-AUTH-1** Регистрация по email + паролю; пароль хранится хэшированным.
- **FR-AUTH-2** Логин выдаёт access-токен (короткий TTL) и refresh-токен.
- **FR-AUTH-3** Обновление по refresh-токену с ротацией.
- **FR-AUTH-4** Отзыв токена / выход.

### Customer (Account)
- **FR-CUST-1** Профиль создаётся при регистрации.
- **FR-CUST-2** KYC-статус: `none → pending → verified | rejected`.
- **FR-CUST-3** Лимиты: дневной лимит на сумму переводов; зависит от KYC-статуса.

### Wallet / Ledger
- **FR-WAL-1** Создание кошелька в выбранной валюте.
- **FR-WAL-2** Просмотр баланса.
- **FR-WAL-3** Пополнение (симулированное).
- **FR-WAL-4** Вывод (симулированный); запрещён при недостатке средств.
- **FR-WAL-5** Любое движение — сбалансированная двойная запись, атомарная в одной транзакции БД.
- **FR-WAL-6** Постинги идемпотентны по `transaction_id`: повтор не создаёт вторую проводку.
- **FR-WAL-7** Баланс не может стать отрицательным (инвариант системы).

### Payments (Orchestrator)
- **FR-PAY-1** Перевод между кошельками: одновалютный и кросс-валютный.
- **FR-PAY-2** Идемпотентность по `Idempotency-Key`: повтор запроса даёт один эффект.
- **FR-PAY-3** Компенсация при частичном сбое (откат уже выполненных шагов).
- **FR-PAY-4** Запрос статуса перевода по его идентификатору.

### Risk (Fraud)
- **FR-RISK-1** Синхронная проверка риска перед списанием.
- **FR-RISK-2** Velocity-лимиты: не более N переводов / сумма за окно времени.
- **FR-RISK-3** Решение `allow | block` с причиной.

### FX
- **FR-FX-1** Возврат курса для валютной пары.
- **FR-FX-2** Конвертация суммы с фиксацией курса на момент перевода.
- **FR-FX-3** Кэш курсов с TTL.

### History (CQRS)
- **FR-HIST-1** Потребление событий леджера и построение денормализованной read-модели.
- **FR-HIST-2** Пагинированная история операций по кошельку.
- **FR-HIST-3** Фильтры по дате и типу операции.

### Notification
- **FR-NOT-1** Потребление событий и отправка уведомления (стаб/лог).
- **FR-NOT-2** Идемпотентная доставка (без повторов при ретрае).

### Audit / Compliance
- **FR-AUD-1** Append-only журнал всех доменных событий.
- **FR-AUD-2** Неизменяемость; запрос по сущности/времени.

### Reconciliation
- **FR-REC-1** Периодический пересчёт балансов из проводок.
- **FR-REC-2** Алерт при расхождении баланса.
- **FR-REC-3** Обнаружение зависших saga (превысивших таймаут) и их пометка.

## 7. Ключевые сценарии

### Сценарий A — кросс-валютный перевод (happy path)
1. Customer вызывает `POST /transfers` с `Idempotency-Key`.
2. Gateway валидирует JWT, проверяет идемпотентность, маршрутизирует в Orchestrator.
3. Orchestrator стартует saga, записывает её состояние.
4. Fraud-проверка (синхронно, gRPC, с таймаутом) → allow.
5. Wallet списывает у отправителя (транзакция БД, блокировка строки).
6. FX конвертирует сумму по зафиксированному курсу.
7. Wallet зачисляет получателю.
8. Orchestrator пишет событие `TransferCompleted` в outbox (в той же транзакции).
9. Relay публикует событие в Kafka; History, Notification, Audit потребляют его независимо.

### Сценарий B — сбой после списания (компенсация)
- Шаги 1–5 как в A; на шаге 7 зачисление падает.
- Saga запускает компенсацию: возврат средств отправителю обратной проводкой.
- Перевод завершается статусом `failed`; публикуется `TransferFailed`.

### Сценарий C — повтор запроса (идемпотентность)
- Тот же `Idempotency-Key` приходит дважды (ретрай клиента).
- Второй запрос возвращает результат первого; вторая saga и вторые проводки не создаются.

### Сценарий D — недоступность Fraud-сервиса
- Fraud не отвечает в пределах таймаута; circuit breaker размыкается.
- Поведение определяется политикой риска (см. NFR-AVAIL): по умолчанию **fail-closed** — перевод отклоняется.

## 8. Нефункциональные требования

> Числовые цели — учебного масштаба (локальный запуск через docker-compose), но трактуются как настоящие SLO для проектирования и тестов.

### Консистентность
- **NFR-CONS-1** Строгая консистентность внутри кошелька: отсутствие отрицательного баланса и lost update при параллельных операциях.
- **NFR-CONS-2** Инвариант денег: Σ debit = Σ credit по каждой транзакции — проверяется в коде и в тестах.
- **NFR-CONS-3** Read-модели (History) — eventual consistency с ограниченным лагом (типично < 5 c).

### Надёжность и доставка
- **NFR-DUR-1** Зафиксированная транзакция не теряется; событие по ней гарантированно публикуется (outbox).
- **NFR-DEL-1** Доставка событий — effectively-once: дедупликация на консьюмере (inbox), retry с backoff, DLQ для «ядовитых» сообщений.
- **NFR-ORD-1** Порядок операций по кошельку сохраняется (партиционирование Kafka по `account_id`).

### Идемпотентность
- **NFR-IDEM-1** Все мутирующие операции идемпотентны (Idempotency-Key на запись, transaction_id на постинг, inbox на консьюмер).

### Доступность и устойчивость
- **NFR-AVAIL-1** Деградация изолирована: падение Notification/History/Audit не влияет на проведение перевода (асинхронность).
- **NFR-AVAIL-2** На каждую синхронную зависимость: timeout, retry с jitter, circuit breaker, bulkhead.
- **NFR-AVAIL-3** Политика риска при недоступности Fraud: по умолчанию fail-closed (отклонять). Конфигурируемо.
- **NFR-AVAIL-4** Graceful shutdown: дренирование in-flight запросов и корректное закрытие консьюмеров.

### Производительность (учебные цели)
- **NFR-LAT-1** p99 перевода (одновалютного) < 300 мс локально.
- **NFR-LAT-2** p99 чтения истории < 100 мс.
- **NFR-THRU-1** Устойчивая обработка ~ нескольких сотен переводов/с локально без роста ошибок.

### Безопасность
- **NFR-SEC-1** JWT на edge; аутентификация сервис-к-сервису внутри (mTLS).
- **NFR-SEC-2** PII (данные KYC) шифруются в покое.
- **NFR-SEC-3** Секреты не в коде/образе; конфиг из окружения.
- **NFR-SEC-4** Rate limiting на edge.

### Наблюдаемость
- **NFR-OBS-1** Сквозной трейсинг каждого запроса через все хопы (OpenTelemetry → Jaeger).
- **NFR-OBS-2** Метрики RED (rate/errors/duration) на каждый сервис; дашборды.
- **NFR-OBS-3** Структурные логи, коррелированные по `trace_id`.
- **NFR-OBS-4** Определены SLO и error budget на ключевые операции.

### Аудируемость / комплаенс
- **NFR-AUD-1** Полная прослеживаемость движения денег: по любой операции восстановим цепочку событий.
- **NFR-AUD-2** Аудит-журнал неизменяем (append-only).

### Масштабируемость и сопровождение
- **NFR-SCALE-1** Сервисы без состояния — горизонтально масштабируемы; состояние — в БД/Kafka.
- **NFR-SCALE-2** БД на сервис; никакого общего стораджа.
- **NFR-MAINT-1** Обратная совместимость API (protobuf) и схем событий; проверяется в CI.
- **NFR-MAINT-2** Миграции БД совместимы с zero-downtime подходом.

## 9. Ограничения и допущения

- **Деньги** хранятся в минимальных единицах (целые числа), никогда во float. У суммы всегда есть валюта.
- **Валюты:** TJS, USD, RUB, EUR (конфигурируемо). *(под подтверждение)*
- **Окружение:** docker-compose; по одному инстансу БД на сервис, один брокер Kafka, локальный Redis.
- **Внешние интеграции** (банки, KYC, FX) — мокаются.
- **Язык/рантайм:** Go 1.22+; gRPC + protobuf; PostgreSQL; Kafka; Redis.

## 10. Архитектурные драйверы (требование → решение)

> Это ядро этапа 1: показывает, почему архитектура именно такая. На собеседовании это умение объяснить «почему» ценится выше знания «что».

| Драйвер (требование) | Решение |
|---|---|
| NFR-CONS-1, FR-WAL-7 (нет отрицательного баланса / lost update) | Ledger в одном владеющем сервисе; транзакции БД; блокировка строки (`SELECT FOR UPDATE`) или сериализуемый уровень. |
| Атомарность между сервисами недостижима | Saga с компенсациями вместо распределённой транзакции. |
| NFR-DUR-1 (событие не теряется при коммите) | Transactional outbox + relay/CDC. |
| NFR-DEL-1 (без потерь и дублей) | Inbox-дедуп на консьюмере + retry-топик + DLQ. |
| NFR-ORD-1 (порядок по кошельку) | Партиционирование Kafka по `account_id`. |
| NFR-IDEM-1 | Idempotency-Key, transaction_id, inbox на трёх уровнях. |
| NFR-AUD-1/2 (аудируемость) | Append-only двойная запись + Audit-сервис как event sink. |
| NFR-AVAIL-2/3 (устойчивость к медленным зависимостям) | Timeout, retry+jitter, circuit breaker, bulkhead; политика fail-closed для риска. |
| NFR-OBS-1 (отладка через 5+ хопов) | OpenTelemetry, проброс trace-context через gRPC и Kafka-заголовки. |
| NFR-LAT-2 (быстрая история) | CQRS: отдельная read-модель, наполняемая из событий. |
| NFR-MAINT-1 (эволюция контрактов) | protobuf + buf + проверка обратной совместимости и schema registry для событий. |

## 11. Определение готовности проекта (success criteria)

Проект считается успешным, когда выполнено и **доказано тестами**:

1. Сквозной перевод работает, включая кросс-валютный.
2. Параллельные переводы с одного кошелька никогда не дают отрицательный баланс или lost update (доказано нагрузочным тестом).
3. Повтор с тем же `Idempotency-Key` даёт ровно один эффект (доказано тестом).
4. Убийство консьюмера на середине потока не приводит к потере или дублю обработки (доказано через inbox/DLQ).
5. Reconciliation обнаруживает искусственно внесённое расхождение баланса.
6. Инъекция латентности в Fraud размыкает circuit breaker; система деградирует по заданной политике (chaos-тест).
7. Полный трейс перевода виден в Jaeger от edge до последнего консьюмера.
